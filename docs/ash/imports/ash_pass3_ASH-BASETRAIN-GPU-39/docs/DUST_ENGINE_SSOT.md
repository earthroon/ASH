# 먼지(DUST) 구조 정리

## 1. 프로젝트 성격
먼지는 범용 챗봇이 아니라 **도메인 특화 변환기**다.  
목표는 초거대 추론이 아니라, **반복이 덜하고 말맛이 덜 깨지는 변환 결과**를 실용 속도로 내는 것이다.

핵심 기준:
- 단순 변환기여야 한다
- 멍청하지 않아야 한다
- 반복 발화가 과도하면 안 된다
- 속도와 운영비가 같이 버텨야 한다
- 실제로 굴릴 수 있는 구조여야 한다

---

## 2. 모델 크기 판단
### 300M
선배가 실제로 강하게 깎아가며 실험한 축.  
속도는 충분했지만, **표현 분포 붕괴와 반복 편향**이 명확하게 드러났다.

관측된 실패:
- 특정 호칭 반복
- 표현 다양성 부족
- 자막기 도메인에서 버릇이 너무 빨리 튐
- 짧은 문장 단위에서 말맛 붕괴가 선명하게 보임

선배 표현으로 요약:
> 여미새가 돼서 언니타령만 함

즉 300M은 **속도는 실전적이지만, 도메인 품질 하한선을 못 넘김**.

### 1.1B
먼지의 실전 최소선.  
300M보다 반복과 표현 붕괴를 넘기기 위한 **최소 유효 크기**로 판단됨.

판단 근거:
- 자막기/변환기 도메인에서는 짧은 출력일수록 반복 편향이 더 선명하게 드러남
- 말맛, 어조, 관계어, 반복 억제가 필요함
- 300M은 이 지점에서 탈락
- 3B는 운영비와 속도 측면에서 과함

### 3B
품질은 올라갈 수 있어도, 먼지의 목적상 **운영상 과한 축**으로 판단.

요약:
- 300M: 빠른데 멍청함
- 1.1B: 안 무너지는 최소선
- 3B: 과투자

---

## 3. 백엔드 철학
먼지는 단순히 모델을 얹은 서비스가 아니라, **실행 계층까지 직접 통제한 엔진**이다.

구성 축:
- Rust
- Vulkan
- WebGPU
- Burn 참조
- CUDA 미사용

중요한 점:
- CUDA가 없어서 우회한 게 아니라, **CUDA 없이도 되는 구조**를 직접 설계했다
- 모델을 돌린 것이 아니라, **GPU가 먹기 좋은 방식으로 연산을 다시 번역했다**

---

## 4. 핵심 최적화 구조

### 4-1. 텐서 텍스처화
텐서를 단순 버퍼가 아니라 **텍스처 자원처럼 취급**해서 계산.

의미:
- GPU가 다루기 좋은 자산 형태로 재배열
- 2D 타일, 샘플링, atlas 구조와 자연스럽게 연결
- 수학 객체를 그래픽스 자산 문법으로 다시 해석한 것

### 4-2. RGBA 채널 역할 분담
RGBA를 단순 색 채널로 보지 않고, 계산 슬롯으로 사용.

- RGB: 실제 값
- A(알파): 체크섬/검증 용도

의미:
- 같은 texel 안에 데이터 채널과 검증 채널을 같이 배치
- 패킹 오류, 블록 꼬임, 로드/스토어 이상을 빠르게 감지할 수 있는 구조

### 4-3. 아틀라스화
흩어진 텐서/블록을 atlas처럼 묶어 배치.

효과:
- locality 개선
- 관리 포인트 감소
- 좌표 접근 구조 단순화
- 타일 연산과 자연스럽게 결합

### 4-4. 텐서코어 모사 8x8
8x8 타일 기반으로 **텐서코어식 연산 감각을 Vulkan 위에서 모사**.

현재 상태:
- 8x8: 실전 운용 SSOT
- 16x16: 실험 브랜치

의미:
- 큰 matmul을 GPU가 먹기 좋은 블록으로 재구성
- 단순 커널 호출이 아니라, 하드웨어 친화적 연산 패턴 설계

### 4-5. 리니어 보간
보간을 linear로 두어 **통과속도 증가**를 노림.

의미:
- 정확도 절대주의보다 도메인에 필요한 정도의 품질 + 전체 throughput을 선택
- 텍스처 샘플링 자산으로서의 이점을 계산에 끌어온 것

## 5. 성능
### 300M 풀학습
- **풀학습 기준 초당 160 tok/s**
- 피크는 **183 tok/s**
- 환경: **RTX 3080**, CUDA 미사용

중요한 해석:
- 느려서 못 쓰는 실험기가 아님
- 엔진 자체는 이미 실전 속도로 버팀
- 300M을 버린 이유는 성능 부족이 아니라 **모델 품질 하한선 문제**였음

요약:
> 300M은 엔진 문제가 아니라 두뇌가 부족해서 탈락

---

## 6. 왜 먼지가 의미 있는가
먼지는 그냥 AI 앱이 아니라:
- 도메인 변환기
- 반복 억제를 고려한 모델 크기 선정
- CUDA 비종속 실행 엔진
- GPU 친화적 데이터 표현
- 학습까지 돌아가는 실전 구조

즉:
- AI를 쓴 것이 아니라
- **AI가 돌아가는 바닥판까지 직접 손댄 것**

---

## 7. 토큰/형태소 구조 실험 브랜치
선배는 토큰화를 단순 문자열 조각이 아니라 **관계 구조**로 보려는 시도를 했다.

아이디어:
- 형태소를 구조텐서처럼 보기
- 초곡면/거리공간처럼 보기
- 관계어, 어미, 호칭, 높임을 구조적 위치로 다루기

이 방향의 의미:
- 반복 편향을 줄일 여지
- 말맛/관계 보존 강화 가능성
- 자막기 같은 도메인 변환기에 특히 잘 맞을 수 있음

현재 판단:
- 철학은 유효함
- 다만 메인 토크나이저를 바로 갈아엎기보다, **보조 구조텐서/거리맵/side embedding** 형태로 실험하는 게 안전함

---

## 8. 토큰 그루핑 아이디어
자주 쓰는 토큰을 hot zone으로 묶어 **GPU locality 개선**을 노리는 발상.

가능한 효과:
- embedding/projection 접근 패턴 단순화
- atlas 안에서 자주 쓰는 토큰군을 가까이 배치
- decode 단계 체감 속도 향상 가능성

주의점:
- 전체 vocab 재배열보다 hot token만 따로 묶는 것이 현실적
- external token id와 runtime packed id는 반드시 분리
- output projection까지 건드릴지 여부는 실험 필요

---

## 9. 공개 전략
먼지는 **AGPL 공개**를 고려 중.

이 선택이 맞는 이유:
- 도구 자체가 실용적임
- 구조적으로도 의미가 있음
- 백엔드 구현 가치가 큼
- 감춰먹는 서버형 독점을 막고, 개선/포크 가능성을 열 수 있음

즉 먼지는:
- 그냥 AI 하나 더
- 그냥 자막기 하나 더
가 아니라,
**대안 실행 구조를 가진 도메인 변환 엔진**으로 소개될 수 있음.

---

## 10. 한 줄 요약
### 사용자 관점
먼지는 **반복이 덜하고 말맛이 덜 깨지는 도메인 변환기**다.

### 기술자 관점
먼지는 **Vulkan/WebGPU/Rust 기반으로 텐서를 텍스처화하고 8x8 텐서코어 모사까지 넣은 비-CUDA 도메인 엔진**이다.

### 선배식 요약
- 300M은 빠른데 멍청했다
- 1.1B는 안 무너지는 최소선이다
- CUDA 안 써도 3080에서 속도는 뽑힌다
- 먼지는 그냥 AI가 아니라, 밑바닥까지 깎은 엔진이다
