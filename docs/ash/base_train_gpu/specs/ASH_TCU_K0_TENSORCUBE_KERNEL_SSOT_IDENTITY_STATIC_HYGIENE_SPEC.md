# ASH-TCU-K0 — TensorCube Kernel SSOT / Identity / Static Hygiene Seal

## Status

`SPEC_COMMITTED_ASH_TCU_K0_TENSORCUBE_KERNEL_SSOT_IDENTITY_STATIC_HYGIENE_SEAL`

## Scope

ASH-TCU-K0 seals TensorCube kernel SSOT, kernel identity, status separation, and static hygiene before any multi-macro widening or performance claim.

This spec is intentionally committed to GitHub as documentation only. Runtime manifests and runtime artifacts are not committed by this spec. They must be generated locally by Rust audit execution.

## Artifact ownership rule

```text
Code bake target: uploaded ASH ZIP tree
Spec ownership: GitHub repository docs
Runtime manifest ownership: local Rust audit output
Runtime artifact ownership: local Rust audit output
```

K0 therefore adds Rust code and tests that can generate the following local outputs when run:

```text
workspace/runtime/tensorcube/ash_tensorcube_k0_kernel_ssot_manifest_latest.json
workspace/runtime/tensorcube/ash_tensorcube_k0_kernel_identity_latest.json
workspace/runtime/tensorcube/ash_tensorcube_k0_static_hygiene_latest.json
workspace/runtime/tensorcube/ash_tensorcube_k0_receipt_schema_latest.json
```

These JSON files are local execution artifacts and must not be pre-baked into the ZIP as committed evidence.

## Rationale

R10-R3 shows that the TensorCube 8x8x4 software matrix kernel has real single-macro math, dispatch, mapped readback digest, and timestamp evidence. The remaining major gap is scaling: the current execution path still behaves as one 16x16 macro tile per dispatch, with `dispatch_workgroups(1,1,1)` and `control_base = 0` style assumptions. Before K3 multi-macro widening, K0 must ensure that the measured kernel identity and static hygiene are sealed.
